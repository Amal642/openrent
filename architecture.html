Data Flows
Whole Project Flow (Simple)
Operator creates accounts and search profiles in the Command Center.
app/workers/account_worker.py runs an account through listing, reply, and reminder phases.
Playwright logs in to OpenRent and reads listings, inbox threads, and landlord replies.
app/db/repository.py loads account persona data; app/ai/personas.py supplies jobs, household, home city, screening posture, and contact-boundary facts.
app/ai/replies.py formats the conversation and calls app/ai/prompts.py to build the OpenAI prompt.
app/ai/prompts.py applies the active design rules. Default fallback is viewing_first_v1; corpus number-capture behavior requires corpus_number_capture_v2.
OpenAI returns one reply; validators remove unapproved phone numbers before the reply is stored.
AI_AUTOSEND=false stores the reply for review only. AI_AUTOSEND=true sends it through app/openrent/messaging.py.
Interactive Session Flow (Human Tester)
Frontend POST /simulation/interactive/start
simulation/interactive.py: start_interactive_session()
simulation/engine/orchestrator.py — setup engine
session_manager.py — emit initial AI message (opening message from design)
Frontend receives session_id + first AI message
Human tester replies → POST /simulation/interactive/{session_id}/message
simulation/policies/production_policy.py — reads _DESIGN_RULES from prompts.py → calls OpenAI
simulation/evaluators/heuristic.py — score turn
Frontend receives updated transcript + state + evaluation
Conversation Design → AI Reply
design_id arrives (from UI or API)
simulation/conversation_designs.py: get_conversation_design() — fetch metadata
app/ai/prompts.py: _DESIGN_RULES[design_id] — fetch rule set
build_reply_prompt(persona, context, rules) — compose full prompt
simulation/policies/production_policy.py — call policy with prompt
OpenAI API → reply text returned
Corpus Number Capture v2 Mode
full-conversations.md is analyzed by scripts/analyze_corpus_number_capture.py; phone redaction comes from scripts/ingest_corpus.py.
corpus_number_capture_v2 is registered in simulation/conversation_designs.py; its actual reply rules live in app/ai/prompts.py.
The mode targets landlord number capture. It does not share the tenant number.
The mode handles refusal: after a landlord refuses phone sharing, the next tenant reply keeps arranging on OpenRent instead of immediately asking again.
Persona templates provide grounded screening facts, including single_income_couple and the landlord_number_boundary style.
simulation/conversation_state.py and simulation/evaluators/heuristic.py measure phone capture, early phone asks, pushed-after-refusal, and viewing progress.

Current live reply flow does not pass conversation_design_id; corpus_number_capture_v2 is not active there unless that wiring is added deliberately.

Key Principles
All reply prompt rules live in app/ai/prompts.py (_DESIGN_RULES). Design metadata, labels, opening messages, and success/failure criteria live in simulation/conversation_designs.py.
corpus_number_capture_v2 is a selectable design, not the default live workflow. Production use requires explicit conversation_design_id wiring plus review of the tenant-mobile safeguard in scripts/process_replies.py.
AI_AUTOSEND=false means generated replies are stored for review only. Live sending happens only when autosend is true and the OpenRent send path succeeds.
Simulation is deterministic — RNG seeding ensures the same scenario + design always produces the same transcript for reproducible testing.
Conversation state analysis uses heuristics only — no LLM call. Fast, cheap, and predictable signal extraction from transcript text.
ProductionPolicy uses temperature=0.0 in simulation mode so evaluations are reproducible across runs.
Repository pattern — all database access goes through app/db/repository.py. No raw queries in business logic.
Event bus captures every step of a simulation session, enabling full replay and Advanced-mode inspection of prompts + completions.
